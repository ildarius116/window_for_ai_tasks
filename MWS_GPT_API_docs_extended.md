# MWS GPT (бывш. LLMOps Platform)

MWS GPT — это облачная B2B-платформа от МТС Web Services (MWS), предназначенная для работы бизнеса с большими языковыми моделями (LLM). Она позволяет использовать, дообучать и внедрять как собственные, так и сторонние (open source) модели, предлагая инструменты вроде чат-интерфейса и no-code конструктора для создания приложений.

**Основные функции и возможности:**

*   **Доступ к LLM:** Платформа объединяет и предоставляет доступ к различным большим языковым моделям, включая собственные разработки МТС и модели с открытым исходным кодом.
*   **Работа с моделями:** Пользователи могут дообучать модели на своих данных и внедрять их в бизнес-процессы через инфраструктуру MWS.
*   **Удобный интерфейс:** Включает чат, который позволяет сохранять и обмениваться запросами (промптами), а также no-code конструктор для создания приложений без привлечения программистов.
*   **Агентская архитектура:** В основе платформы лежит агентская архитектура, которая позволяет моделям самостоятельно выбирать необходимые навыки для решения конкретной задачи.
*   **Гибкая оплата:** Стоимость зависит от количества использованных токенов, что делает платформу доступной для компаний разного размера, включая малый и средний бизнес.
*   **Инфраструктура:** Работает на облачной инфраструктуре МТС, что обеспечивает отказоустойчивость и масштабируемость, и планируется возможность развертывания на серверах заказчика.

**Наш сайт:** [https://mws.ru/services/mws-gpt/](https://mws.ru/services/mws-gpt/)

### 6.4. Анализ задержки запуска Flow в Langflow

**Контекст:** Между нажатием кнопки запуска пайплайна в UI и началом выполнения компонентов на бэкенде наблюдается задержка 10+ секунд.

#### 6.4.1. Общая схема выполнения
1.  **FRONTEND:** Нажатие кнопки -> `sendMessage()` -> `buildFlow()` (валидация) -> `POST /build/{flowId}/flow` -> получение `job_id` -> `GET /build/{jobId}/events`
2.  **BACKEND:** `start_flow_build()` -> создание очереди -> `build_graph_and_get_order()` (запрос flow из БД, `Graph.from_payload()`, `initialize_run()`, сортировка) -> кэширование графа -> `build_vertices()` (выполнение)

#### 6.4.2. Выявленные проблемы

##### 1. Граф пересоздается с нуля при каждом запуске (влияние: 2-5 сек)
*   **Проблема:** Кэш (`chat_service.set_cache`) не проверяется перед построением.
*   **Решение:** Реализовать проверку кэша перед построением, использовать `Graph.update()` для инкрементального обновления, добавить версионирование flow.

##### 2. Deep copy всех данных flow (влияние: 100-500 мс)
*   **Проблема:** `copy.deepcopy(flow_object)` всей JSON-структуры.
*   **Решение:** Заменить на shallow copy с точечным копированием, использовать `msgspec`/`orjson`, выполнять `process_flow` один раз при сохранении.

##### 3. Последовательное инстанциирование компонентов (влияние: 500-3000 мс)
*   **Проблема:** Для каждой вершины последовательно вызывается `instantiate_component()`.
*   **Решение:** Кэширование скомпилированных классов, параллельное инстанциирование (через `ThreadPoolExecutor`), предварительная компиляция.

##### 4. Двухфазный HTTP-запрос на фронтенде (влияние: 100-300 мс)
*   **Проблема:** Два последовательных запроса: `POST /build/...` и `GET /build/.../events`.
*   **Решение:** Использовать режим `DIRECT` (возвращает streaming-ответ сразу), WebSocket-соединение, параллельная инициализация.

##### 5. Загрузка flow из БД (большой JSON-блоб) (влияние: 50-200 мс)
*   **Проблема:** Вся структура flow хранится как единый JSON-блоб.
*   **Решение:** In-memory кэширование `flow.data`, сжатие данных при хранении.

##### 6. Инициализация сервиса трассировки (влияние: 0-500 мс)
*   **Проблема:** При каждом запуске инициализируются трассировщики (LangSmith и др.).
*   **Решение:** Ленивая инициализация (один раз при старте), асинхронная инициализация в фоне.

### 6.5. Как работают связи компонентов в воркфлоу Langflow и почему они ломаются при обновлении версии

#### 6.5.1. Как устроены связи компонентов

*   **Модель данных (Edge):** JSON, содержит сериализованные `sourceHandle` (выход) и `targetHandle` (вход). Handle сериализуются в строку через `scapedJSONStringify`.
*   **Хранение в ноде:** Полный снимок определения компонента на момент сохранения (`template`, `outputs`, `code`).
*   **Хранение в БД:** Flow хранится в таблице `flow` как JSON-колонка `data` (содержит `nodes` и `edges`). Бэкенд возвращает сырой JSON.
*   **Создание связи:** `onConnect()` -> `isValidConnection()` проверяет совместимость типов -> создается edge.
*   **Загрузка flow:** Фронтенд загружает JSON -> `processFlows()` -> `processFlowEdges()`, `processFlowNodes()` -> `resetFlow()` -> `detectBrokenEdgesEdges()`, **`cleanEdges()`** (удаляет невалидные) -> рендер.
*   **Валидация на фронтенде (`cleanEdges()`):** Для каждого edge парсит handle и сравнивает строки с "ожидаемым" handle, собранным из текущего состояния ноды. Удаляет при несовпадении, а также edge-и к скрытым (`show === false`) или `advanced` полям.
*   **Валидация на бэкенде:** При сборке графа валидирует по текущим определениям компонентов из кода. Несовпадение типов -> `ValueError`.

#### 6.5.2. Где ломается при обновлении

1.  **Строковое сравнение в `cleanEdges()`:** Сравнивает строки вместо совместимости типов. Любое изменение `input_types`, `type`, `output_types` ломает связь.
2.  **Output не найден по имени:** Если `output` переименован или удален, а `selected_output` указывает на старое имя, edge удаляется.
3.  **Несовпадение `output_types` в сериализованном handle:** Сохраненный edge кодирует типы на момент создания.
4.  **Скрытые поля удаляют edge-и:** Если `show` становится `false`, edge-и к полю удаляются.
5.  **Валидация на бэкенде при запуске:** Финальная проверка типов может отклонить edge.
6.  **Это происходит "тихо":** `cleanEdges()` удаляет edge-и без уведомления, `detectBrokenEdgesEdges()` может их не заметить.
7.  **Это необратимо:** Auto-save сохраняет flow без связей в базу.

#### 6.5.3. Найденные баги

1.  **`cleanEdges()` сравнивает строки вместо проверки совместимости типов.**
2.  **Шаблоны не обновляются до очистки edge-ей:** `cleanEdges()` валидирует по старым данным нод.
3.  **Auto-save делает потерю edge-ей необратимой.**
4.  **`detectBrokenEdgesEdges()` и `cleanEdges()` по-разному ищут source output:** Разная логика поиска и наличие проверки `tool_mode` только в `cleanEdges()`.
5.  **`selected_output` ссылается на удаленный output:** Все edge-и от компонента удаляются.

#### 6.9.6. RBAC в Langflow с использованием Casbin

##### 6.9.6.1. Что сейчас есть
*   Простая авторизация: `is_superuser=True` (полный доступ) или `False` (только свои ресурсы).
*   Зависимость `CurrentActiveUser` используется в 76 местах.

##### 6.9.6.2. Выбранный подход
*   **Dependency Injection Override:** Подмена `get_current_active_user` без изменения кода роутеров.
*   **Casbin:** Зрелая библиотека для RBAC с доменами (проектами).

##### 6.9.6.3. Два варианта реализации

| Критерий | Вариант A (MVP) | Вариант Б (полный) |
|---|---|---|
| Ролевая модель | 5 фиксированных ролей, привязка к Folder | Организации, проекты, редактируемые роли, группы |
| Новых таблиц в БД | 2 (`user_project_role`, `casbin_rule`) | 8 (+`organization`, `rbac_project`, `role_definition`, `user_group` и др.) |
| Файлов Langflow изменено | 1 (`main.py`) | 2 (`main.py`, `router.py`) |
| Редактируемые роли | Нет | Да |
| Группы пользователей | Нет | Да |
| Рекомендация | **Начать с Варианта А** | Расширить до Варианта Б позже |

##### 6.9.6.4. Вариант А (MVP) - детали
*   **Таблица `user_project_role`:** user_id, folder_id, role_name.
*   **Конфиг Casbin:** Модель с `sub`, `dom` (folder_id), `obj`, `act`.
*   **EndpointMapper:** Маппинг URL -> resource_type и action (по HTTP методу).
*   **ResourceResolver:** Определение `folder_id` по ресурсу (flow, project и др.).
*   **Dependency Overrides:** Подмена `get_current_active_user`, `get_current_active_superuser`.
*   **Точка интеграции:** 7 строк в `main.py` при старте (если включено переменной окружения).
*   **API назначения ролей:** `/api/v1/rbac/projects/{folder_id}/members` (GET, POST, PATCH, DELETE).

##### 6.9.6.5. Вариант Б (полный) - что добавляется
*   Доп. таблицы: `organization`, `rbac_project`, `role_definition` (с JSON-правами), `user_group` и др.
*   Расширенный конфиг Casbin: поддержка `org_admin`, групп (`g3`).
*   **RoleManager:** Синхронизация `role_definition` с политиками Casbin.
*   **GroupManager:** Каскадное назначение ролей группам.
*   API управления RBAC: 5 роутеров (организации, роли, члены проекта, группы, импорт).

*   **Вариант А:** `main.py` (+7 строк), `pyproject.toml` (+2 зависимости).
*   **Вариант Б (доп.):** `router.py` (+1 строка).

##### 6.9.6.9-6.9.6.12. Риски, плюсы/минусы, миграция
*   **Плюсы подхода:** Ноль изменений в роутерах, минимальные конфликты при мерже, feature-flag, изоляция кода RBAC.
*   **Минусы:** Нужно поддерживать маппинг эндпоинтов, `api_key` и `variable` остаются owner-only в MVP.
*   **Миграция с A на Б:** Без потери данных, можно делать по шагам.

#### 6.9.7. Анализ кодовой базы по ролевке LF и оценка предлагаемых решений

*   **Текущее состояние:** FastAPI, SQLModel, JWT. Примитивная авторизация (`is_superuser`). Зависимость `CurrentActiveUser` в 76 местах.
*   **Четыре подхода к интеграции RBAC:**
    1.  **Dependency Injection Override (РЕКОМЕНДОВАН):** Ноль изменений в коде Langflow, полная изоляция, минимальные риски.
    2.  **Middleware-based:** Централизованная проверка, но возможны merge конфликты в `main.py` и сложность парсинга путей.
    3.  **Plugin Architecture (Fork):** Высокие риски конфликтов при мерже, нужно изменять ~30 файлов.
    4.  **Service Layer Wrapper (Proxy):** Операционная сложность, дополнительная задержка.
*   **Почему Casbin:** Разработчики Langflow планируют использовать Casbin. Поддерживает RBAC с доменами, PostgreSQL, async, высокую производительность.
*   **Интеграция с Casbin:** Требуется кастомизация (ScopedEnforcer, RoleManager, GroupManager) для поддержки иерархии "Организация-Проект-Ресурс", редактируемых ролей и групп.

*   **Как сделать RAG?** Документация Langchain или No-code конструктор Langflow.
*   **Как сделать агента?** Документация Langchain или No-code конструктор Langflow.
*   **Почему запрос обрабатывается так долго?** Скорость ответа (TPS) зависит от длины генерируемого текста. Используйте асинхронную генерацию (streaming) для улучшения UX.
*   **Модель лежит, я получаю ошибку APITimeoutError: Request timed out, что делать?** Запрос мог попасть в очередь, если модель перегружена (параметр concurrency). Используйте streaming. SLA сервиса - 99% (планируется 99.95%).
*   **Как посмотреть доступность платформы?** В Grafana: [https://grafana-gbp.genai.mts-corp.ru/login](https://grafana-gbp.genai.mts-corp.ru/login) (reader/reader).
*   **Кто может проконсультировать по внедрению LLM в мой продукт?** ЦК Диджитализации процессов.
*   **Где посмотреть полезный контент по внедрению LLM?** Записи вебинаров на главной странице Confluence.
*   **Как понять, какая LLM локальная, а какая внешняя?** В UI есть признак "Внутренняя/Внешняя". В API этого параметра нет для совместимости с OpenAI.
*   **Как понять, какие LLM можно использовать с чувствительной информацией?** Модели с признаком "Внутренняя" запускаются на серверах МТС, данные никуда не уходят.
*   **У меня техническая проблема/нужна консультация.** Создать заявку по шаблонам (Консультации, Инциденты, Сервисные запросы).
*   **У меня ключ старого формата (ASGK), что делать?** Обменять на новый формат через `curl http://api.llmops.mts-corp.ru/_asgk_exchange -H "X-ASGK-TOKEN: <your asgk>"`.
*   **У меня еще остались вопросы, где задать?** Telegram-чат или MTS LINK Чаты.
