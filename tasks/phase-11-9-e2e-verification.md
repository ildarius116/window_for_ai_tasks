# phase-11-9 — End-to-end верификация Phase 11

## Цель
Пройти живые smoke-сценарии в OpenWebUI, убедиться, что pptx-тракт работает, файлы скачиваются и открываются, fallback-пути не дают 500.

## Что сделать
Подготовка:
- `docker compose up -d --build pptx-service openwebui auto-router` (bootstrap сам переупсёртит функции).
- Убедиться, что `OWUI_ADMIN_TOKEN` установлен в `.env` и проброшен.
- Проверить health: `docker compose exec openwebui curl -s http://pptx-service:8000/health`.

Сценарии (запускать руками в UI через «MWS GPT Auto 🎯»):

1. **Презентация из резюме (основной путь).**
   - Приложить `Сабиров Ильдар - Python-разработчик.pdf`.
   - Написать: «Вот резюме. Сделай из него презентацию.»
   - Ожидаем: ответ с короткой подводкой + `📎 [....pptx](...)`.
   - Скачать файл, открыть в LibreOffice → титул + 5–8 слайдов с буллетами из резюме.

2. **Презентация из головы (без вложения).**
   - «Сделай презентацию про Python async/await на 5 слайдов.»
   - Ожидаем: файл с 6 слайдами (титул + 5), буллеты релевантны теме.

3. **DOCX-источник.**
   - Приложить любой `.docx` (например, сгенерированный из markdown).
   - «Сделай из этого презентацию.»
   - Ожидаем: работает так же, как с PDF.

4. **doc_qa не ломается.**
   - Приложить PDF, спросить: «Что написано в этом документе?»
   - Ожидаем: обычный doc_qa ответ, НЕ презентация.

5. **Fallback — pptx-service выключен.**
   - `docker compose stop pptx-service`.
   - Повторить сценарий 1.
   - Ожидаем: в чате текстовая структура слайдов (markdown), без 500.
   - `docker compose start pptx-service` в конце.

6. **Fallback — OWUI_ADMIN_TOKEN отсутствует.**
   - Временно убрать токен из env openwebui, `docker compose up -d openwebui`.
   - Повторить сценарий 1.
   - Ожидаем: в ответе summary с предупреждением + текстовый markdown-план (без ссылки).
   - Вернуть токен.

7. **Ручной выбор модели не триггерит pptx.**
   - Сменить в dropdown на `mws/qwen3-235b`, написать «Сделай презентацию про Python».
   - Ожидаем: обычный текстовый ответ, роутер bypassed.

8. **Лимит размера.**
   - Приложить файл > 20 МБ.
   - Ожидаем: fallback с сообщением «Файл слишком большой для автоматической презентации».

Логирование:
- Для каждого сценария зафиксировать `trace_id`, решение классификатора и итог (PASS/FAIL) в отчёте `tasks_done/phase-11-done.md`.

## Критерии готовности
- Сценарии 1–4 и 7 — PASS.
- Сценарии 5, 6, 8 — graceful fallback без 500.
- Никаких регрессов в smoke-группах A/B/C/E из Phase 9/10 (быстрый прогон одного представителя каждой группы).
